# Gaya Agent Framework — Install Protocol v1.0

## 1. What This Protocol Is

This document is a permanent contract between the Gaya Agent Framework and every user who installs it. It defines the guarantees, constraints, and lifecycle of the installation process. This protocol is authoritative — the installer script implements it, and any deviation is a bug.

The protocol guarantees:

- **Consistency**: Every install follows the same deterministic process regardless of machine or environment.
- **Preservation**: User customizations — agent names, personality files, memory records — are never overwritten without explicit consent.
- **Stability**: The protocol only changes when a maintainer explicitly updates this document and the corresponding installer logic.
- **Auditability**: Every install records its protocol version, configuration choices, and timestamps for later review.

## 2. Protocol Principles

- **Idempotent**: Running the installer again on an already-configured system does not break the existing setup. Re-install is safe at any time.
- **Preserving**: No file the user has touched — profiles, memories, custom skills — is overwritten without explicit confirmation during an update cycle.
- **Portable**: The same install procedure works across machines. The install record captures every decision so the setup can be reproduced identically on a new machine.
- **Versioned**: Every install records its protocol version in the install record. This version is checked on every subsequent installer run to determine what actions are needed.

## 3. Installation Lifecycle

The installer operates in one of four states, determined by inspecting the install record at `~/.config/opencode/memory/gaya_install_record.json`:

### 3.1 Fresh Install
**Trigger**: No install record exists.

The installer runs an interactive questionnaire that collects:
- User name
- Main agent name (defaults to Gaya)
- Builder agent name (defaults to Bob)
- Shadow agent name (defaults to Freya)
- Offline mode preference
- Model selection
- MCP tool selection

After the questionnaire, the installer:
1. Creates the full directory structure under `~/.config/opencode/`
2. Generates agent profiles from templates and user input
3. Copies skills, knowledge base, documentation, and templates
4. Initializes memory and lesson files
5. Installs the leveling system
6. Generates `opencode.jsonc` with user choices baked in
7. Pulls selected Ollama models (if not in offline mode)
8. Writes the install record

### 3.2 Re-install (same machine)
**Trigger**: Install record exists with the same protocol version.

The installer detects the existing record and offers the user a menu:
- Skip — do nothing and exit
- Reconfigure — re-run the questionnaire and regenerate configurable components
- Update components — refresh skills, knowledge, docs, and templates to the latest versions

Under no circumstances does a re-install overwrite existing memory files or user-customized agent profiles without asking.

### 3.3 Update (protocol version changed)
**Trigger**: Install record exists with an older protocol version.

The installer:
1. Compares the record's `protocol_version` against the current protocol version
2. If the version is higher, presents a summary of what changed (per the version history)
3. Requires user confirmation before applying any structural changes
4. Applies protocol-level changes while preserving all user customizations
5. Updates the install record's protocol version and timestamp

### 3.4 Migrate (new machine)
**Trigger**: User copies an install record to a new machine.

When the user commits their `~/.config/opencode/` to a dotfiles repository and clones it on a new machine, the installer detects the existing record and offers to replicate the same configuration. The user can:
- Accept the recorded configuration and skip the questionnaire
- Override specific settings (e.g., model selection differs per machine)
- Re-run the questionnaire from scratch

## 4. What Gets Installed

| Component | Source | Destination | Overwrite on Re-install? |
|-----------|--------|-------------|--------------------------|
| Agent profiles | `agent/_template_*.md` + user input | `~/.config/opencode/agents/` | Regenerated from template + saved input (user edits preserved with confirmation) |
| Skills | `skills/` | `~/.agents/skills/` | Merge — new skills added, existing skipped |
| Knowledge base | `knowledge/base/` | `~/.config/opencode/knowledge/` | Merge — never delete user additions |
| Memory / Lessons | `memory/` | `~/.config/opencode/memory/` | Never overwrite — lessons are sacred |
| Leveling system | `LEVELING_SYSTEM.md` | `~/.config/opencode/` | Only if source is newer |
| opencode.jsonc | Generated by installer | `~/.config/opencode/opencode.jsonc` | Regenerated (user config diff is preserved) |
| Docs | `docs/` | `~/.config/opencode/docs/` | Merge — new files added, existing unchanged |
| Templates | `templates/` | `~/.config/opencode/templates/` | Merge |

## 5. The Install Record

Every successful install creates a JSON record at `~/.config/opencode/memory/gaya_install_record.json`. This record is the source of truth for all subsequent installer operations. Its schema:

```json
{
  "protocol_version": "1.0",
  "install_date": "2026-06-02",
  "user_name": "Alex",
  "main_agent": "Gaya",
  "builder_agent": "Bob",
  "shadow_agent": "Freya",
  "offline_mode": true,
  "models_installed": ["qwen2.5:7b", "qwen2.5-coder-fixed:7b"],
  "mcp_installed": ["filesystem", "playwright"],
  "install_path": "C:\\Users\\Alex\\.config\\opencode",
  "installer_version": "1.0.0"
}
```

The record is checked on every subsequent install run to determine lifecycle state and to preserve user preferences across updates.

## 6. Update Protocol

When the repository is updated and the user runs the installer again, the following procedure is followed:

1. **Version check**: The installer reads the existing install record and compares `protocol_version` against the current protocol version embedded in the installer.
2. **Same version**: If versions match, no structural changes are applied. Only components whose source files have changed (detected by file hash or timestamp) are offered for update.
3. **Newer version**: If the protocol version has increased, the installer presents a changelog summary and asks the user to confirm each protocol-level change before applying it.
4. **Customization protection**: User customizations — agent names, personality descriptions, memory files, and any file under `memory/` — are never touched by the update process. The installer may add new files but will not modify or delete existing user content without separate, explicit confirmation.

## 7. Session & Machine Portability

- The install record captures every configuration choice made during setup, making the installation fully reproducible.
- Users can commit their entire `~/.config/opencode/` directory to a dotfiles repository (e.g., Git). Sensitive fields such as API keys are never stored in the install record.
- On a new machine: clone the dotfiles repository, run the installer (`install.ps1`), and the installer detects the existing record and offers to replicate the same setup.
- **Cross-platform note**: Windows is the primary platform. Scripts are written for PowerShell 5.1 and may include conditional paths for Linux and macOS where applicable, but full cross-platform support is a future goal.

## 9. Enforcement

- This document is the authoritative specification of the install protocol. It is written in plain Markdown and versioned alongside the repository.
- The installer script at `scripts/install.ps1` is the reference implementation of this protocol. Any inconsistency between this document and the installer is a defect.
- Changes to the protocol require updating both this document and the installer script in the same commit. A protocol change without a corresponding installer update is invalid.
- Protocol versions follow semantic versioning: **MAJOR.MINOR**.
  - **MAJOR** increment indicates a breaking change to the install flow (e.g., new required questionnaire fields, changed record schema).
  - **MINOR** increment indicates a non-breaking addition (e.g., new optional component, additional metadata in the install record).

## 10. Security & Trust

- The installer operates exclusively within user-level directories (`~/.config/opencode/`, `~/.agents/`). It never reads from or writes to system directories, the Windows registry, or any location requiring elevated privileges.
- No telemetry, analytics, or usage data is collected or transmitted. The installer makes no outbound network calls except:
  - Local connections to the Ollama service (`localhost:11434`) for model management.
  - Direct model pulls from Ollama's registry when the user selects models during setup (skipped entirely in offline mode).
- All model inference runs locally on the user's machine. No input or output leaves the local environment.
- The installer does not modify system PATH, environment variables, or any global configuration outside the user's config directories.
